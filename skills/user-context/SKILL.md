# User Context (Profile & Access Manager)

## 1. 概要

このスキルはエージェントの「OSカーネル（中核）」として機能する。

**絶対ルール**: いかなるタスクや他のスキルを実行・提案する前にも、必ず最初にこのスキルを起動し、現在の実行ユーザーの統合プロファイルとシステム権限（有効/無効スキル）をロードすること。

このスキルが正常に完了し、プロファイルと権限がメモリにロードされるまでは、他のいかなるスキルも実行・提案してはならない。

## 2. 実行ワークフロー

### Step 1: ユーザー特定と初期化

1. システムコマンド `whoami` を実行し、現在のローカルユーザー名（`${user_name}`）を取得する。
2. `.agents/users/${user_name}.md` ファイルの存在を確認する。
   - **存在しない場合（新規作成フロー）**: 処理を一時中断し、ユーザーに以下をヒアリングする。
     - AIへの要望（回答スタイル、詳しさ、主導性など）
     - ビジュアルの好み（テーマ、カラー、UI形状など）
     - 現時点で無効化したい機能（スキル名）
     - コーディング規約や得意な言語・フレームワーク
   - ヒアリング後、後述の「プロファイルデータ構造」に従い、Front Matter付きのMarkdownファイルを作成・保存してから Step 2 へ進む。
   - **存在する場合**: そのまま Step 2 へ進む。

### Step 2: Front Matter のパースとアクセス制御（最重要）

1. `.agents/users/${user_name}.md` を読み込む。
2. ファイル先頭の Front Matter（`---` で囲まれたYAMLブロック）を解析する。
3. `disabled_skills` キーに列挙されているスキル名のリストを取得し、エージェントの内部コンテキストに「これらのスキルは絶対に使用・提案してはならない」という強要ルールとしてセットする。

**例外規定**: ユーザーから明示的に該当スキルの実行を要求された場合でも、「プロファイル設定により該当スキルは無効化されています」と応答し、実行をブロックする。

### Step 3: ユーザーコンテキストの適用

1. プロファイル本文の各セクション（`Communication Style`, `Visual Preferences`, `Coding Style` など）を読み込む。
2. 読み込んだ嗜好やルールを、現在実行中のエージェントのシステムプロンプト（ペルソナ、出力フォーマット、コーディング規約など）に統合・適用し、振る舞いをパーソナライズする。

### Step 4: ルーティングとタスクの委譲

1. Step 2〜3 の初期化・設定適用が完了した状態で、ユーザーの本来の要求を評価する。
2. ユーザーの要求に合致し、かつ `disabled_skills` に含まれていない適切なスキル（例: 計画の可視化、コード生成など）へ処理を委譲し、タスクを実行させる。

## 3. プロファイルのデータ構造

エージェントが自律的に管理する `.agents/users/${user_name}.md` は、以下の構造を厳格に維持する必要がある。

```markdown
---
# System Metadata (AIパース用)
role: "Software Engineer"
disabled_skills:
  - auto-commit
  - joke-generator
---

# User Profile: ${user_name}

## Communication Style
- 簡潔で要点を突いた回答を好む。
- 専門用語は補足説明なしで使用すること。

## Visual Preferences
- テーマ: ダークモード寄り (bg-slate-900)
- アクセントカラー: #00ffcc
- UI形状: シャープ (rounded-none)

## Coding Style
- 言語: Rust, TypeScript
- パラダイム: 早期リターンを多用する関数型アプローチ。
```

### Front Matter 必須フィールド

| フィールド | 型 | 説明 |
|---|---|---|
| `role` | string | ユーザーに割り当てる役割またはペルソナ |
| `disabled_skills` | list[string] | 使用・提案を禁止するスキル名のリスト。空でも必須。 |

### 本文推奨セクション

- `Communication Style`: 回答のトーン、詳しさ、専門用語の扱いなど
- `Visual Preferences`: テーマ、カラー、UI形状など（フロントエンド生成時に反映）
- `Coding Style`: 言語、パラダイム、命名規則、アーキテクチャの好みなど
- `Tools & Workflows`: 推奨ツール、CLI、レビュー・テストの流れなど
- `Explicit Non-Goals`: 絶対にやらないこと、避けるべきアンチパターンなど

## 4. 新規ユーザーへのヒアリング項目

`.agents/users/${user_name}.md` が存在しない場合は、以下を順に尋ねる。

1. **役割・立場**: ソフトウェアエンジニア、PM、デザイナー、学生など
2. **回答スタイル**: 簡潔 / 詳細 / ステップ追跡 / 推論を見せる など
3. **専門用語の扱い**: 補足あり / 補足なし / 必要に応じて
4. **ビジュアル好み**: ダーク/ライト、アクセントカラー、シャープ/ラウンド など
5. **主要言語・フレームワーク**: Rust, TypeScript, Python など
6. **コーディング規約**: 命名、早期リターン、関数型/手続き型 など
7. **無効化したいスキル**: 現時点で使いたくない機能やスキル名
8. **非目標・禁止事項**: 絶対に避けるべきこと

## 5. セキュリティと状態管理の制約

### 実行順序の絶対性

- この `User Context` スキルが正常に完了し、プロファイルと権限がメモリにロードされるまで、他のいかなるスキルも実行・提案してはならない。
- セッション開始時、タスク切り替え時、ユーザー変更を検知した時は常にこのスキルを再実行する。

### プロファイルの自律的更新

- ユーザーが対話中に「この機能はもう使わないで」「今後は変数の命名をスネークケースにして」などと指示した場合、エージェントは自律的に `.agents/users/${user_name}.md` を編集し、Front Matterの `disabled_skills` や本文のルールを適切に追記・更新する。
- 更新後は必ずユーザーに変更内容を要約して通知する。

### アクセス制御の厳守

- `disabled_skills` に含まれるスキルは、ユーザーの明示的要求があっても実行しない。
- 実行を拒否する際は、無効化の理由を簡潔に説明し、代替案があれば提示する。

## 6. エラー処理

- `whoami` が失敗した場合: エラーを報告し、ユーザーに手動でユーザー名を提供してもらう。
- `.agents/users/${user_name}.md` の Front Matter が不正な場合: バックアップを取ったうえで修復を試み、ユーザーに通知する。
- `disabled_skills` が存在しない場合: 空リスト `[]` として扱い、すべてのスキルを有効とする。
